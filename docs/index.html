01 · Architecture Globale

Pattern : Microservices Event-Driven
avec Bounded Contexts DDD

Une plateforme PSP de ce niveau ne peut pas être un monolithe. L'architecture est découpée en domaines métier indépendants (DDD Bounded Contexts), chaque service Spring Boot owning sa propre base de données, communicant via événements Kafka.

Principe clé — Database per Service : Chaque microservice Spring Boot possède exclusivement sa propre base de données. Payment Service → sa PostgreSQL. Fraud Engine → sa base ML. Merchant Service → sa PostgreSQL. Ils ne partagent jamais une base commune. La communication entre services passe UNIQUEMENT par les événements Kafka ou les APIs REST. C'est ce qui garantit l'indépendance de déploiement et la scalabilité.

Acquirer Connector

Abstraction des rails bancaires locaux. Un adapter par partenaire (CMI Maroc, SATIM Algérie, MonétiqueTunisie). Gère les retries, timeouts, et la normalisation des codes de réponse ISO 8583.

Comptabilité en partie double (double-entry). Chaque transaction génère deux écritures comptables (débit/crédit). Source of truth financière. Immuable — aucune écriture ne peut être modifiée.

Transfer Service (C2C)

Transferts compte-à-compte entre marchands ou vers wallets. Utilise le pattern Saga (Choreography) pour coordonner la déduction chez l'émetteur et le crédit chez le destinataire de façon atomique via Kafka.

Pattern Saga — Transactions distribuées sans 2PC
// PaymentSagaOrchestrator.java — Saga avec compensation en cas d'échec @Service public class PaymentSagaOrchestrator { // Étape 1 : Vérification fraude // Étape 2 : Réservation fonds (débit provisoire) // Étape 3 : Appel acquéreur // Étape 4 : Écriture ledger // Si Étape 3 échoue → compensation : libérer les fonds réservés public PaymentResult execute(PaymentCommand cmd) { var sagaId = UUID.randomUUID().toString(); try { fraudEngine.check(cmd); // Step 1 ledgerService.reserve(cmd.amount(), sagaId); // Step 2 var acquirerResponse = acquirerConnector.authorize(cmd); // Step 3 ledgerService.commit(sagaId, acquirerResponse); // Step 4 eventPublisher.publish(new PaymentCompletedEvent(sagaId, cmd)); return PaymentResult.success(acquirerResponse.authCode()); } catch (AcquirerException e) { ledgerService.rollback(sagaId); // Compensation automatique eventPublisher.publish(new PaymentFailedEvent(sagaId, e.code())); throw e; } } }
⚠️
Idempotence — Critique en FinTech : Si un client clique deux fois "Payer" ou si un réseau relance la requête, la même transaction ne doit être exécutée qu'une seule fois. La solution : chaque requête de paiement transporte un idempotencyKey unique. Spring Boot stocke ce key dans Redis (TTL 24h). Si la key existe déjà → retourner le résultat existant sans re-exécuter.

04 · Moteur Fraude & Risque

Fraud Engine — Scoring Temps Réel

Chaque transaction est scorée en moins de 150ms avant d'être autorisée. Un score trop élevé déclenche un blocage ou une friction supplémentaire (OTP SMS).

🧠

Rule Engine

Règles métier configurables en temps réel sans redéploiement : montant max par jour, pays bloqués, vélocité (N transactions en M minutes), heure suspecte, terminal inconnu. Stockées en base + cache Redis.

Drools / Easy Rules Redis Cache Hot-reload rules Admin UI config
🤖

ML Scoring Model

Modèle ML (XGBoost ou Random Forest) entraîné sur l'historique transactions. Score 0.0 → 1.0. Features : montant, heure, fréquence, géolocalisation, device fingerprint, historique marchand. Servi via API REST Python.

Python Flask/FastAPI scikit-learn / XGBoost Feature Store Model versioning
⚡

Velocity Check

Détecte les patterns de fraude par vélocité : trop de transactions en peu de temps, même montant répété, même terminal avec cartes différentes. Implémenté avec compteurs Redis TTL (sliding window).

Redis Sliding Window Lua Scripts Redis Kafka Streams Real-time counters
🗺️

Device & Geo Intelligence

Analyse du device fingerprint, adresse IP, géolocalisation. Détecte les VPN/Tor, les IP blacklistées, les changements de pays impossibles en peu de temps (impossible travel). Base GeoIP MaxMind.

MaxMind GeoIP2 Device Fingerprinting IP Reputation Tor/VPN detection
// FraudScoringService.java — Pipeline de scoring <150ms @Service public class FraudScoringService { public FraudScore score(PaymentContext ctx) { var signals = CompletableFuture.allOf( // Parallélisation pour tenir <150ms CompletableFuture.supplyAsync(() -> ruleEngine.evaluate(ctx)), // ~20ms CompletableFuture.supplyAsync(() -> velocityCheck.check(ctx)), // ~15ms (Redis) CompletableFuture.supplyAsync(() -> mlModel.predict(ctx)), // ~80ms (HTTP FastAPI) CompletableFuture.supplyAsync(() -> geoIntelligence.analyze(ctx)) // ~30ms ).thenApply(v -> new SignalAggregator().combine( ruleScore, velocityScore, mlScore, geoScore )); double finalScore = signals.join().getWeightedScore(); return switch (finalScore) { case d when d < 0.3 -> FraudScore.ALLOW; case d when d < 0.7 -> FraudScore.CHALLENGE; // → OTP SMS default -> FraudScore.BLOCK; }; } }
05 · Conformité Réglementaire

KYC · KYB · AML · PCI-DSS

Une plateforme PSP en Afrique du Nord doit se conformer aux régulations locales (Bank Al-Maghrib, Banque d'Algérie, BCT) et aux standards internationaux.

KYC

Know Your Customer

Vérification identité utilisateur final. CNI/Passeport scan + vérification OCR, selfie liveness check, score de confiance. Intégration API d'identité nationale si disponible.

KYB

Know Your Business

Onboarding marchand : RC, IF, ICE (Maroc), documents légaux. Vérification statut entreprise, bénéficiaires effectifs (UBO), screening OFAC/World-Check.

AML

Anti-Money Laundering

Screening continu transactions contre listes sanctions internationales (OFAC, EU, ONU). Détection patterns blanchiment (structuring, layering). Rapport GAFI si dépassement seuils.

PCI

PCI-DSS Level 1

Si traitement données cartes : chiffrement AES-256 au repos, TLS 1.3 en transit, tokenisation via Vault, audit logs immuables, tests de pénétration annuels, segmentation réseau.

RGPD

Protection Données

Droit à l'oubli, minimisation données, consentement explicite, chiffrement données personnelles, audit trail accès, notification breach dans les 72h.

PSP

Agrément Régulateur

Bank Al-Maghrib (Maroc), Banque d'Algérie, BCT (Tunisie). Rapports réglementaires périodiques, limites transactions, reporting BAM en temps réel au-delà des seuils.

// AuditService.java — Audit trail immuable pour compliance PCI-DSS @Service public class AuditService { // Chaque action sensible génère une entrée immuable // Stockée dans une table PostgreSQL append-only (pas d'UPDATE/DELETE autorisés) // + envoyée vers Elasticsearch pour recherche compliance public void record(AuditEvent event) { var entry = AuditEntry.builder() .eventId(UUID.randomUUID()) .timestamp(Instant.now()) .actorId(event.actorId()) // Qui a fait l'action .actorType(event.actorType()) // MERCHANT / ADMIN / SYSTEM .action(event.action()) // LOGIN / VIEW_TRANSACTION / EXPORT_DATA .resourceId(event.resourceId()) // Sur quelle ressource .ipAddress(event.ip()) .userAgent(event.userAgent()) .result(event.result()) // SUCCESS / FAILURE .checksum(computeHmac(event)) // Intégrité cryptographique .build(); auditRepository.save(entry); // PostgreSQL append-only elasticsearchClient.index(entry); // Elasticsearch pour search compliance } }

// WebhookDeliveryService.java — Livraison fiable avec retry @Service public class WebhookDeliveryService { @Retryable(maxAttempts = 5, backoff = @Backoff(delay = 1000, multiplier = 2.0)) public void deliver(WebhookEvent event, Merchant merchant) { var payload = objectMapper.writeValueAsString(event); var signature = hmacSha256(payload, merchant.getWebhookSecret()); var response = httpClient.post(merchant.getWebhookUrl()) .header("X-Signature", "sha256=" + signature) .header("X-Event-Type", event.type()) .body(payload) .execute(); if (!response.is2xxSuccessful()) { throw new WebhookDeliveryException("HTTP " + response.statusCode()); } webhookLogRepository.save( WebhookLog.success(event.id(), response.statusCode(), Instant.now()) ); } @Recover public void onMaxRetryExceeded(WebhookDeliveryException ex, WebhookEvent event) { webhookLogRepository.save(WebhookLog.failed(event.id(), ex.getMessage())); alertingService.notifyMerchantWebhookFailure(event); } }

09 · Monétisation

Moteur de Facturation PSP

Chaque transaction génère une commission. Chaque marchand paie un abonnement. Le moteur de facturation doit être exact au centime et auditable.

💸

Commission Transactionnelle

Pourcentage + fixe par transaction réussie. Exemple : 1.5% + 2 MAD. Calculé en temps réel lors du paiement, enregistré dans le Ledger comme écriture de commission. Settlement hebdomadaire vers compte PSP.

📦

Abonnements Marchands

Plans mensuels/annuels : Starter / Growth / Enterprise. Facturation automatique via le Billing Service le 1er du mois. Paiement par prélèvement du solde marchand ou carte.

🔌

API Usage Billing

Facturation à l'usage pour les intégrateurs API : par requête au-delà d'un quota inclus, par webhook livré, par rapport généré. Compteurs Redis incrémentés à chaque appel.

// FeeCalculationService.java — Calcul commission en temps réel @Service public class FeeCalculationService { public FeeResult calculate(Transaction tx, MerchantPlan plan) { // Commission = (montant * taux%) + fixe // Taux dépend du plan marchand et du canal (QR = moins cher que carte) var rate = plan.getRateFor(tx.channel(), tx.cardBrand()); var percentageFee = tx.amount().multiply(rate.getPercentage()); var fixedFee = rate.getFixedAmount(); var totalFee = percentageFee.add(fixedFee); // Plafond si configuré if (plan.hasFeeCapFor(tx.channel())) { totalFee = totalFee.min(plan.getFeeCap(tx.channel())); } // Écriture comptable : débit compte marchand, crédit compte PSP ledgerService.bookFee(LedgerEntry.builder() .debit(tx.merchantAccountId(), totalFee) .credit(pspRevenueAccount, totalFee) .reference(tx.id()) .build()); return new FeeResult(totalFee, rate, tx.amount().subtract(totalFee)); } }
